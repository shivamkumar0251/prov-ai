Prov AI's **Data Security Framework** ensures that sensitive information remains safe while empowering users to access the insights they need. 

---

## Security Components

Prov AI provides two primary layers of data restrictions:

- **Row-Level Security (RLS)**: Restricts which rows of a model (table) each user can see based on their identity and context (e.g., region, org ID).
- **Column-Level Security (CLS)**: Controls which columns are visible to different groups, protecting sensitive attributes like salaries or PII.

### Session Properties
Session properties are the "keys" to security. They are contextual values (like `user_id`, `department`, `role`) that drive the evaluation of your policies at query time.

---

## Enforcement Surfaces

Prov AI applies security controls across all interaction points:

| Surface | Row-Level Security | Column-Level Security |
| :--- | :--- | :--- |
| **Asking Questions** | Yes | Yes |
| **Generating Charts** | Yes | Yes |
| **Data Previews** | Yes | Yes |
| **API Requests** | Yes | Yes |
| **Spreadsheets** | Yes | Yes |
| **Dashboards** | Not Yet | Not Yet |

---

## How it Works: AI-Generated SQL

Even when SQL is generated by AI, Prov AI routes all queries through the **Prov Engine**. This ensures that policies are applied *before* the query is executed against your data source.

### Preventing SQL Injection
Session properties are never injected as raw strings into your warehouse. They are validated and safely bound during the Prov Engine's parsing and dialect-SQL generation phase, ensuring consistent, least-privilege access.

---

## Common Scenarios

- **Multi-Tenant Isolation**: Each customer sees only their own data.
- **Regional Access**: Regional managers only see rows belonging to their territory.
- **Sensitive Field Masking**: Hide PII (SSN, Email) from non-HR users.
- **Owner Access**: Sales reps only see deals they own.
